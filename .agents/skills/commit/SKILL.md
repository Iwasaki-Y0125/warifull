## ゴール
- 「今の変更内容」に沿ったコミットメッセージ（日本語）を作る
- ユーザー指定のPRテンプレに沿ってPR本文を作る
- 指定Issueがあれば `Closes #NN` で紐づける

## 作業手順（必須）
1. `git status` を確認し、未ステージの変更がある場合はステージングを促す
2. `git branch --show-current` で現在ブランチを確認し、`feature|fix|chore|refactor|docs/issue-<num>-...` 形式なら Issue番号を自動抽出する
3. `git diff --staged` を読み、変更点を要約する
4. 出力は.md形式で以下の順で行う:
  A. コミットメッセージ案（1案）
  B. PR本文（指定テンプレ。不要セクションは削る。` ```md ` で囲む）
  C. 末尾に必ず `Closes #NN`（ユーザー指定がある場合）

## 出力フォーマット

### A. Commit message
- 1案のみ

### B. PR本文（md形式で出力）
```md
## 目的
-
## 結論
-
## 変更点
-
## 動作確認
-
## 参考資料（1次ソース）
-
## 関連Issue
Closes #NN
```

## 制約
- diffから読み取れないことは断定しない（不明点は「不明」と書く）
- 不要なセクションは削る
- コミットメッセージとPR本文は必ず日本語で出力する
- Issue番号の自動抽出ルール:
  - 現在ブランチが `feature|fix|chore|refactor|docs/issue-<num>-...` に一致する場合は `<num>` を関連Issueとして使う
  - ユーザーがIssue番号を明示した場合は、明示値を最優先する
  - 抽出できない場合は推測せず、Issue番号の明示入力を求める
- Commit messageの先頭のプレフィックスは以下から選ぶ:
  - `feat:` 新機能追加（ユーザーが使える機能が増える）
  - `fix:` バグ修正
  - `refactor:` 振る舞いを変えない整理・分割・置き換え
  - `docs:` ドキュメントのみ変更
  - `chore:` 環境整備・依存関係・ビルド/CI・Dockerなど（機能は変えない）
  - `WIP:` 進行中（完成していない）
- PR本文のセクションタイトルは必ず上記テンプレに従う
- ()での補足説明は原則使わない
